# CI Design: DataTalk AI

## Objective

Create a GitHub Actions continuous-integration pipeline for the DataTalk AI
MVP. The pipeline validates backend, frontend, and Docker configuration on
every push and pull request without deploying the application.

## Scope

- Continuous integration only.
- No cloud deployment, production credentials, or environment secrets.
- One workflow with independent jobs for backend, frontend, and Docker.
- Update the README to show CI status and remove CI/CD from future work.

## Triggers and Concurrency

- Run on every `push` and `pull_request`.
- Use a concurrency group based on workflow and Git reference.
- Cancel an older in-progress run when a newer commit arrives on the same
  branch or pull request.

## Permissions

- Set workflow permissions to `contents: read`.
- Do not grant write permissions or expose repository secrets.

## Backend Job

- Runner: `ubuntu-latest`.
- Python: 3.12.
- Cache pip packages using `requirements.txt` as the dependency key.
- Install `requirements.txt`.
- Compile `app` and `alembic` with `python -m compileall`.
- Run `python -m pytest -q`.
- Use the mock LLM provider and no external API keys.

## Frontend Job

- Runner: `ubuntu-latest`.
- Node.js: 20.
- Cache npm packages using `frontend/package-lock.json` as the dependency key.
- Install dependencies with `npm ci`.
- Run `npm run typecheck`, `npm run test`, and `npm run build`.

## Docker Job

- Runner: `ubuntu-latest`.
- Validate the checked-in Compose definition with `docker compose config`.
- Do not start containers because runtime integration is outside this CI slice.

## README Changes

- Add a GitHub Actions status badge for the CI workflow.
- State that GitHub Actions validates backend, frontend, and Docker.
- Remove CI/CD from the list of future roadmap work.

## Failure Behavior

- Each job reports its own failure independently.
- A failure in one job does not prevent the other jobs from producing evidence.
- GitHub determines overall workflow success only when all three jobs pass.

## Acceptance Criteria

- The workflow YAML is syntactically valid.
- Backend, frontend, and Docker commands match local validated commands.
- The workflow contains no secrets or write permissions.
- `git diff --check` reports no whitespace errors.
- Existing backend and frontend test suites remain green locally.
